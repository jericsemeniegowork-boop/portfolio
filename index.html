Scope

  • Login and session handling
  • Request creation and editing
  • Submission for approval
  • Approval and rejection flow
  • Repository search, filter, sort, and export

  • Approved request remains editable
  • Wrong approver receives the request
  • Mandatory fields are not enforced
  • Repository list shows stale status

Bug Report: Approved Request Remains Editable

Summary

An approved purchase request remains editable by the requester after final approval.

Steps to reproduce

  1. Log in as requester.
  2. Open an already approved request.
  3. Attempt to edit protected fields such as amount or justification.
  4. Click Save.

Expected result

Approved request should be read-only. Protected fields should not be editable.

Actual result

Requester is able to edit protected fields after approval.

Severity / Priority

High / High

Business impact

This can affect approval integrity, audit accuracy, and downstream reporting.

Evidence note

Issue reproduced through controlled test execution. No confidential screenshot is included in this representative sample.

UAT Sample: Order-to-Invoice Flow

Purpose

Validate that a business transaction can move from order creation to invoice confirmation with correct status updates and records.

Flow

  1. Create order using valid customer and item data.
  2. Confirm order details and pricing.
  3. Process delivery or fulfillment step.
  4. Generate invoice.
  5. Validate final status and report output.

Expected result

The transaction completes without blocking errors, statuses update correctly, and the final invoice record matches the source order details.

Postman-style API Validation

Endpoint: GET /api/purchase-requests/{request_id}

Checks

  • Response code is 200.
  • Request ID matches the requested value.
  • Status field exists.
  • Status value is valid.
  • Restricted internal fields are not exposed to requester role.
pm.test("Status code is 200", function () {
  pm.response.to.have.status(200);
});

const data = pm.response.json();

pm.test("Request ID matches", function () {
  pm.expect(data.requestId).to.eql(pm.variables.get("request_id"));
});

pm.test("Status is valid", function () {
  pm.expect(["Draft", "Pending Approval", "Approved", "Rejected"])
    .to.include(data.status);
});

SQL Validation Checks

Representative queries used to confirm whether UI actions were stored correctly.

SELECT request_id, status, updated_at
FROM purchase_requests
WHERE request_id = 'PR-10025';

SELECT request_id, action, actor, action_time
FROM purchase_request_history
WHERE request_id = 'PR-10025'
ORDER BY action_time DESC;

SELECT request_id, COUNT(*) AS conflicting_rows
FROM purchase_requests
WHERE request_id = 'PR-10025'
  AND status IN ('Approved', 'Rejected')
GROUP BY request_id;

Selenium / Python Sample

Representative automation check for approved request read-only behavior.

from selenium.webdriver.common.by import By

def test_approved_request_is_read_only(driver, login):
    login("approver_user")
    open_request(driver, "PR-10025")
    approve_request(driver)

    login("requester_user")
    open_request(driver, "PR-10025")

    amount = driver.find_element(By.ID, "amount")
    assert amount.get_attribute("disabled") in ("true", "disabled")
